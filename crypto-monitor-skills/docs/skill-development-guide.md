# Skill development guide — гайд по развитию skills

## Введение

Skills в этом каталоге — продакшн-ресурс банковской системы. Любые изменения должны проходить через ревью и регрессионное тестирование. Этот гайд описывает: как создавать новые skills, как модифицировать существующие, как поддерживать качество.

## Принципы

### 1. Один skill — одна задача

Не пытайтесь сделать «универсальный» skill, который и классифицирует, и реферирует. Большие монолитные skills хуже триггерятся, сложнее тестируются и сложнее в поддержке. Если задача делится на этапы — делайте отдельные skills.

### 2. «Pushy» description

Description в YAML-фронтматтере — это то, по чему Claude решает, использовать skill или нет. Без явной директивы Claude недотриггерит. Правильные обороты:

- «Use this skill whenever…»
- «Triggers include phrases such as: '…', '…', '…'»
- «Use ALSO when…»

Неправильные обороты («Helps with…», «Can be used for…») приводят к тому, что skill не активируется.

### 3. Императив в инструкциях

Тело SKILL.md — повелительное наклонение: «Извлеки», «Сформируй», «Не используй». Это снижает двусмысленность интерпретации.

### 4. Объясняйте причину правил

Вместо
> «MUST NOT use quotes longer than 15 words»

пишите
> «Не используй цитат длиннее 15 слов: это нарушает копирайт источников и создаёт юридический риск для банка-получателя сводки.»

Модель лучше следует правилам, когда понимает их назначение.

### 5. Примеры и антипримеры

Каждый skill должен содержать минимум:
- 2 разных по типу примера input → output;
- 1 антипример с пояснением, почему он неправильный.

Примеры — это закодированные best practices. Модель тренируется на них в контексте.

### 6. Объём SKILL.md

Стремитесь к 200–500 строк тела SKILL.md. Крупные справочники, глоссарии, шаблоны — выносите в `references/` и `assets/` и ссылайтесь оттуда. Это даёт Progressive Disclosure: модель подгружает только то, что нужно.

## Структура каталога нового skill

```
your-new-skill/
├── SKILL.md          ← обязательно
├── references/       ← опционально, для справочных материалов
│   ├── *.md
├── assets/           ← опционально, для шаблонов и образцов
│   ├── *.html
│   ├── *.json
├── examples/         ← опционально, для развёрнутых input → output
│   ├── *.json
└── evals/            ← обязательно
    └── evals.json
```

## Шаблон SKILL.md

Заполняйте пропуски сверху вниз. Каждый раздел обязателен (если только не отмечен как опциональный).

## Evals: как писать тесты

Каждый skill должен иметь минимум 3 теста в `evals/evals.json`. Цель — регрессионное тестирование: при изменении SKILL.md мы должны быстро понять, не сломали ли поведение.

### Шаблон evals.json

```json
{
  "skill_name": "your-skill-name",
  "evals": [
    {
      "id": 1,
      "name": "happy-path-case",
      "prompt": "<полный текст user-сообщения>",
      "expected_output": "<описание ожидаемого результата>",
      "assertions": [
        "Output is valid JSON",
        "<проверяемое утверждение>"
      ]
    }
  ]
}
```

### Что должно быть в evals

- **1 счастливый путь** — типичный успешный случай.
- **1 граничный случай** — пустой вход, очень короткий вход, неоднозначный вход.
- **1 кейс, проверяющий специфическое правило** — например, что копирайт-ограничение соблюдается.

### Assertions

Пишите assertions в простых формулировках:
- «Output is valid JSON»
- «field X is present»
- «field X equals Y»
- «field X is in array [...]»
- «string Z does not appear in output»

Раннер evals (см. `integration-guide.md`) интерпретирует assertions либо автоматически (для простых проверок), либо вручную ревьюером.

## Версионирование

Используйте SemVer-подобную схему в поле `version` фронтматтера:

- **MAJOR** (1.x.x → 2.0.0) — несовместимые изменения контракта (изменился формат выхода, добавлены обязательные поля входа).
- **MINOR** (1.0.x → 1.1.0) — новые возможности, обратно-совместимые правила.
- **PATCH** (1.0.0 → 1.0.1) — исправления в тексте, уточнения примеров, без изменения поведения.

Каждое изменение фиксируется в разделе «Версионирование» в SKILL.md:

```markdown
## Версионирование

- **1.1.0** — добавлено поле `confidence` в выход.
- **1.0.1** — уточнён пример 2.
- **1.0.0** — первая версия.
```

## Жизненный цикл изменения

1. **Идея.** Запрос от заказчика, баг от получателя сводки, регрессия из мониторинга.
2. **Ветка в Git.** `feature/improve-classifier-confidence` или `fix/summarizer-15-word-limit`.
3. **Изменение SKILL.md** и/или references/, assets/.
4. **Обновите evals.json:** добавьте кейс, который ранее не покрывался.
5. **Бамп версии** в фронтматтере.
6. **Запуск evals** — все кейсы должны пройти.
7. **Pull Request с ревью** минимум одного коллеги.
8. **Merge** в main; pipeline пересобирает образ системы.
9. **Деплой в STAGE → PROD** по обычной CI/CD-процедуре.

## Создание нового skill: чек-лист

- [ ] Назвал в kebab-case, начинается с `crypto-` для consistency.
- [ ] YAML-фронтматтер: name, description, version, license.
- [ ] Description — pushy, с триггерными фразами.
- [ ] Назначение — 2–3 предложения.
- [ ] Когда использовать — конкретные сценарии с фразами.
- [ ] Контракт: входная JSON-схема и выходная JSON-схема.
- [ ] Алгоритм — шаги в повелительном.
- [ ] Правила — с пояснением «почему».
- [ ] Минимум 2 примера input → output.
- [ ] Минимум 1 антипример.
- [ ] Граничные случаи.
- [ ] Версионирование (1.0.0).
- [ ] Все ссылки на references/ ведут на существующие файлы.
- [ ] evals/evals.json: минимум 3 кейса, включая один граничный.
- [ ] Skill самостоятелен (можно вызывать без других skills).
- [ ] Обновлён MANIFEST.json в корне.

## Антипаттерны

### 1. Слишком общий description

❌ `description: Helps with crypto tasks.`

Skill не будет триггерить. Будь конкретен.

### 2. Зависимость от других skills внутри SKILL.md

❌ В body: «См. skill X для дополнительной обработки».

Skills должны быть независимы. Если нужна цепочка — это работа оркестратора, не skill.

### 3. Огромный SKILL.md (1000+ строк)

❌ Всё-в-одном, включая большие справочники.

Выноси в references/ и ссылайся.

### 4. Размытые правила

❌ «Учитывай контекст и принимай разумные решения».

Будь конкретен: что считать контекстом, что считать разумным.

### 5. Отсутствие evals

❌ skill без evals — невозможно тестировать регрессии.

Минимум 3 кейса всегда.
