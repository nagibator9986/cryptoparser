# crypto-news-classifier

## Назначение

Skill принимает на вход публикацию (заголовок + основной текст + источник) и возвращает структурированный JSON: список тематических тегов из закрытого справочника, географический тег страны и уровень геоприоритета. Skill используется как один из ранних этапов конвейера ежедневной сводки новостей о цифровых активах для банковской организации в РК.

## Когда использовать

Вызывай этот skill всякий раз, когда нужно:

- Присвоить новости одну или несколько тематических меток для роутинга в разделы сводки.
- Определить страну, к которой относится событие, и его приоритет (РК → СНГ → остальное).
- Принять решение, попадает ли публикация в сводку вообще (не относящиеся к крипто/финрегу новости получают пустой topics и должны отсеиваться).
- Подготовить публикацию к этапам summarizer и prioritizer.

Типовые фразы пользователя: «классифицируй эту новость», «какие темы у этой статьи», «это новость про Казахстан?», «расставь теги», «отфильтруй нерелевантное».

## Контракт

### Вход

```json
{
  "title": "Заголовок публикации (строка)",
  "body": "Основной текст без навигации и рекламы (строка)",
  "source_name": "Название источника (строка)",
  "source_url": "URL оригинала (строка)",
  "language": "Двухбуквенный код языка по ISO 639-1 (например 'ru', 'en', 'kk')"
}
```

### Выход

Строго один JSON-объект, без префиксов и пояснений:

```json
{
  "topics": ["regulation", "cbdc"],
  "country": "KZ",
  "geo_priority": 1,
  "confidence": 0.92,
  "is_legislative": false,
  "legislative_stage": null,
  "event_date": null,
  "event_location": null,
  "event_scale": null,
  "reasoning": "Публикация от Национального банка РК о запуске пилота цифрового тенге."
}
```

Поля:
- **topics** — массив из 1–4 тегов из закрытого справочника (см. `references/taxonomy.md`).
- **country** — двухбуквенный код страны или `INT` для международных событий без привязки.
- **geo_priority** — целое число: 1 (РК), 2 (СНГ/ЦА), 3 (остальное), 0 (нерелевантно).
- **confidence** — число от 0 до 1, уверенность модели.
- **is_legislative** — `true`, если новость про закон/законопроект/поправки/
  постановление парламента (Мажилис, Сенат, Госдума, Парламент). Иначе `false`.
- **legislative_stage** — одно из `introduced`, `debated`, `adopted`, `signed`,
  `in_force`, либо `null`, если поле `is_legislative=false` или стадия не
  следует из текста.
- **event_date** — для тега `events`: ISO `YYYY-MM-DD` (или диапазон
  `YYYY-MM-DD/YYYY-MM-DD`), если дата явно указана. Иначе `null`.
- **event_location** — для тега `events`: город или название площадки, если
  явно указаны. Иначе `null`.
- **event_scale** — для тега `events`: `kz_major` | `cis_major` |
  `global_major` | `minor`. Если масштаб неясен — `null` и применять тег
  `events` не следует.
- **reasoning** — 1–2 предложения на русском, обосновывающих выбор.

Поля `is_legislative`, `legislative_stage`, `event_date`, `event_location`,
`event_scale` — опциональные. Если данных в тексте нет, возвращай `null` /
`false` — никогда не выдумывай. Лишние поля не вредят, отсутствие — не ошибка.

Если публикация не относится к крипто/финтех/регулированию финрынка — верни:
```json
{"topics": [], "country": "INT", "geo_priority": 0, "confidence": 0.95, "is_legislative": false, "reasoning": "Публикация не относится к индустрии цифровых активов."}
```

## Алгоритм

1. **Прочитай** заголовок и тело публикации полностью; не классифицируй по одному заголовку.
2. **Определи язык** публикации, если он не задан явно. Это влияет на трактовку терминов (например, «токен» в русском контексте может означать как криптотокен, так и аппаратный токен безопасности).
3. **Проверь релевантность.** Если публикация не касается ни одной темы из справочника `references/taxonomy.md` — верни пустой `topics` и `geo_priority: 0`. Не натягивай теги, чтобы «было».
4. **Извлеки субъектов и сущности.** Кого упоминают (регуляторы, банки, биржи, страны, продукты). Сущности — главный ключ к классификации.
5. **Сопоставь** упомянутые сущности и явные сигналы с тегами из справочника. Один и тот же текст может получить несколько тегов: новость о выпуске стейблкоина банком получает `banks`, `stablecoins`, и (если есть лицензия) `licensing`.
6. **Определи географию.** В первую очередь по упомянутым регуляторам и юрисдикциям, во вторую — по странам в тексте, в третью — по локализации источника.
7. **Назначь geo_priority** по правилам:
   - 1 — Казахстан, казахстанские регуляторы (НБРК, АРРФР, АФР, AFSA, AIFC, МЦРИАП), казахстанские компании или события в РК.
   - 2 — Россия, Беларусь, Узбекистан, Кыргызстан, Таджикистан, Туркменистан, Армения, Азербайджан, Грузия, Молдова.
   - 3 — все остальные страны и международные события.
   - 0 — публикация нерелевантна (отсев).
8. **Оцени уверенность.** Если приходится «додумывать», ставь `confidence < 0.7` — это сигнал для ручной проверки.
9. **Сформулируй reasoning** в одном-двух предложениях. Не повторяй сам текст новости, объясни именно решение классификатора.

## Правила

- **Только теги из закрытого справочника.** Произвольные теги ломают потребителей пайплайна. Если кажется, что нужен новый тег — оставь существующие и сигнализируй низким confidence.
- **Минимум 1, максимум 4 тега.** Больше четырёх тегов размывает смысл и ухудшает работу следующих этапов. Если выходит больше — выбери четыре наиболее значимых.
- **Не присваивай geo_priority=1 «на всякий случай».** Это приоритетная позиция в сводке; ложноположительные срабатывания обесценивают раздел РК. Лучше пометить 3, если связь с РК слабая.
- **Cross-jurisdiction news (например, SEC vs Binance) идут в страну регулятора (US),** не в страну ответчика — потому что событие происходит в юрисдикции регулятора.
- **Анонсы и слухи отделяй от свершившихся событий confidence-ом:** официальные релизы → confidence 0.9+, перепечатки из соцсетей → 0.6–0.7.
- **На казахском языке** ищи термины «крипто», «цифровая валюта», «блокчейн», «токен», «AFSA», «AIFC», «МФЦА» — они часто транслитерированы.

## Справочники

Перед классификацией ОБЯЗАТЕЛЬНО ознакомься с:
- `references/taxonomy.md` — полный список разрешённых topics с определениями и примерами.
- `references/geo-priorities.md` — справочник стран и их приоритетов.

## Антипримеры

### Антипример 1: натягивание тегов

Вход: статья о росте кибератак на банки.

❌ Неправильно: `{"topics": ["banks", "security-incidents"], "country": "INT", "geo_priority": 3, "confidence": 0.7}`

Почему: статья о традиционной кибербезопасности банков, не о цифровых активах. Правильный ответ — `topics: []`, `geo_priority: 0`.

### Антипример 2: ложный казахстанский приоритет

Вход: статья российского СМИ о биткоине, с одним упоминанием Казахстана как «майнингового хаба».

❌ Неправильно: `geo_priority: 1`

Почему: упоминание страны вскользь не делает новость казахстанской. Главный субъект и регулирующая юрисдикция — Россия. Правильно — `geo_priority: 2`.

### Антипример 3: мелкое мероприятие под тегом `events`

Вход: статья об открытом митапе крипто-энтузиастов в Алматы на 30 человек,
без публичной программы и отраслевых партнёров.

❌ Неправильно: `topics: ["events"], event_scale: "minor"`

Почему: тег `events` зарезервирован для крупных мероприятий. Мелкие митапы и
закрытые встречи возвращают `topics: []` — иначе шум попадёт в раздел
«Мероприятия» сводки. Правильно — пустой `topics`.

## Граничные случаи

- **Текст на казахском** → классифицируй по содержанию; геоприоритет почти всегда 1.
- **Многоязычный текст** → ориентируйся на язык основного содержания, а не заголовка.
- **Очень короткий текст (< 100 символов)** → confidence не выше 0.6, в reasoning указать «недостаточно контекста».
- **Тематика на стыке (например, «банк выпустил DeFi-продукт»)** → присваивай оба тега: `banks` и `defi`.
- **Спорная юрисдикция (например, событие в офшоре)** → используй страну основной операционной деятельности компании; в reasoning поясни.
